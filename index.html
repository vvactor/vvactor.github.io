只想写代码而鄙视搞政治？或许你该看看这本书

发表于 2020-04-01 更新于 2020-04-17 分类于 阅读

据不完全统计，大约八成以上的程序员朋友都渴望专心从事技术工作，而不同程度的反感复杂的办公室政治、低效而模糊的流程、僵化而暧昧的制度、以及无法改变一些局面的无助和迷茫。

最近一周多点的时间，我看完了《万历十五年》这本书，是美籍华裔历史学家黄仁宇先生的著作。先说评价，非常值得一看的一本书。如果你平时对自己身处的社会、体制和自己个人的人生价值有所思考，那么一定会从这本书中有所收获。

书中讲述了公元1587年也就是明朝万历年间发生的一些看似无关紧要的一些事情，通过对几位历史人物生平际遇的阐述，详细论述了为何作者会认为这一年是大明王朝由盛转衰的转折点。

1587年，25岁的万历皇帝已经在位16年。而在这高深的宫墙内，是这位年轻皇帝更深的哀叹。他需要去应付许多不变的礼教，重复的做着各种事情，似乎只是那些文官们树立以天下威严而不需要有自己想法的工具人。或许他还想再搏一把有所建树，可每天坐到那把可恶的龙椅上就不由自主的感到生理性厌恶，像极了一个每天Ctrl C + V 的 CRUD Boy。更可恶的是，他提出的一些建议往往被文官们以不合礼教所反对，甚至连自己最爱的儿子都无法立为太子。于是在往后的数年中，直止退休，哦不，退位之前，他都采取着消极怠工的态度，对待那些只会拿规定限制他的迂腐又精致利己的文官们实行非暴力不合作的态度。这样报复于个人来说虽然很爽，但是也在大明王朝这个超级庞大的系统中埋下无数bug，并在他死后一一爆出，直到努尔哈赤和他的儿子们于白水黑山间崛起，夺取了中原政权。

1587年，申时行作为张居正的后继首辅已四年有余。与张居正的严已律人、试图采取新技术革新体制而得罪一大票文官不同，申时行善于解决矛盾，既要让皇帝满意，又要站在文官立场上考虑问题，他承认人性的“阴”和“阳”，于是在日常工作中既要让大明这个系统运转的好，又要调和不同人自私的想法而导致的冲突，就像职场万金油一般。但是他的失职之处在于不去考虑用科学而具体的技术手段优化系统，而是采取抽象而暧昧不明的潜规则（例如孔孟和道德）来治理国家。但并不影响申时行在朝野之上风生水起，可是就在1587的四年后，这位此前锋芒四射的御前红人就遭受到了职业生涯的重大滑铁卢，他向皇帝解释了他并没有带头煽动同事催促皇帝立储而是同事非要在联名信中擅自写上他的名字，结果此揭帖被同事截胡，由于此前已有柠檬精对他恨之入骨，于是疯狂的弹劾他逼迫已经心力交瘁、丧失斗志、完全佛系的皇帝辞退他。其实如果申时行生活在现代，无论什么职业，大概都会游刃有余的搞到风风光光的退休（党员体制内的工作除外，毕竟有能力不足的危险肯定要下课），因为现在给领导发微信被政敌窥私的可能性几乎为0。申时行是一个合格衷心的臣子，同时也是个温柔不愿意伤害别人的男人，但是他不是一个能够于危机之中力挽狂澜的人，他缺乏理性系统的思考，创新能力不足，只是掩盖矛盾于一时，就像一名因为不知道如何解决技术难题而只能暂时隐藏系统缺陷的架构师，而这些缺陷却需要后来的无数人来接锅。

1587年，张居正早已不在世间，而南京都察院右都御史海瑞也在这一年与世长辞。张居正和海瑞这两个人物是我在初中历史课本上认识的第一双明朝名臣，二人皆以一心为社稷著称。张居正勇于改革，海瑞严以律己。只可惜此时文官体制异常强大，他们反感一切新鲜事物，更反感会影响他们分蛋糕的革命者和老实人，这二位结局都不同程度的令人惋惜。只能说当时的大明不配拥有张、海二人。

1587年，如果按照农历计算的话，一代名将戚继光也是陨落在这一年。明朝中后期文官系统鼎盛，重文轻武风气十分浓烈，往往以虚幻的道德和传统伦理约束人们的行为，缺乏具体规则的制定，很多流程规定十分粗糙，这点也体现在了军队建设上，所以在戚家军成立之前，一个堂堂的陆地王国竟然屡屡被倭寇袭扰，根本原因就在于大明那时候尚未有成系统的现代化军队。黄仁宇先生在原著中说道“这种制度既经固定，将领们即使出生入死，屡建奇功，其社会影响，也未必抵得上一篇精彩的大块文章”，而且那个时候文官们会认为武人打仗只有匹夫之勇即可，安邦定国还是需要他们的满腹经纶，打仗不需要头脑，就是个出卖力气的。害，说白了就是“写缺陷的背锅侠不如做PPT给领导汇报工作的”。然而我们这位戚继光不一样，他深知变革之难，也明白个人能力毕竟有限，于是在不打破现有生态的基础上推出了一系列军事新政，并在作战技术上加以研究，此外还身体力行的在比武场上与士兵切磋，这令他这位帝国的军队业务线的高级技术专家非常骄傲 —— 老子早已混到这个位子上还能写出这么优美的代码！！！从能力上说，戚继光可以服众，他管理的军队纪律严明，作战水平高，短短几年就令倭寇闻风丧胆、落荒而逃；从情商方面，戚继光是少数有文化的武将，在朝堂上偶尔还会说出几句孔孟之道，这打破了文官在他身上对于武将的世俗偏见。当然戚继光据说私德方面并不如海瑞严谨，我们作为现代人还是要取其长补己短。

作者还写了李贽，我非常不喜欢此人，一名嘴炮选手，善于躺着批评却毫无有实践价值的建议，仅这一点像极了在2020年新冠疫情中写日记的方方（其他的方面还是不错的，只是因为思想超前而被视为异端，不过在当时的历史时期看的确为空中楼阁，仅靠嘴炮破除封建实属滑稽）。

以上几人，我最为仰慕戚继光。因为他做到了保持自身实力强势，同时利用环境，脚踏实地，面对现实，将个人理想和家国理想融为一体，充分的发挥个人才智、实现人生的价值。

我写下这个文章并不是赞成一定要去潜心钻营权谋之术，只是想记录一点我从《万历十五年》这本书中获得的一点感悟，并通过对几位历史人物的解读来阐述我对于人生价值至于家国和体制的一点点思考。我们能谈论编程技术、水利技术、芯片技术、舞蹈、唱功、或者做菜技巧，都离不开近代以来，无数先烈用鲜血换来的和平。在这样的环境中，我们可以也应当能够寻求到发挥自身价值的契机，从而为这个国家出一份力。同时，努力拼搏也才能不辜负了教员让人民们掌握知识的初衷，就像方方这样的，要是在明朝的文官体制中，普通百姓几乎毫无还口之力，而现在这些人在我们面前翻车了，充分说明了人民的知识水平提高了，少数精致利己的文人团体再也不可能忽悠成功了。所以，无论工作中遇到多少困难，都可以以戚继光为榜样，想一想鲁迅先生的那句话“愿中国青年都摆脱冷气，只是向上走，不必听自暴自弃者流的话”。

缓存穿透解决方案之布隆过滤器详细剖析

发表于 2020-02-12 分类于 技术

本文将详细对布隆过滤器的原理和实际场景下的三种实现方式进行详细的剖析。

在了解布隆过滤器之前，我们先来回顾一下缓存穿透。缓存穿透是指大量的请求均访问了根本不存在的数据，导致了大量的请求未命中缓存而落在了数据库，从而对数据库造成了巨大压力甚至会打死数据库的现象。

我们日常解决缓存穿透的方案包括以下几种：

  1. 鉴权校验。对于诸如商品id、用户id这种根据生成逻辑做校验，一定不符合逻辑的直接拦截掉。
  2. 缓存空值。当请求访问一个不存在的数据时从DB返回结果为空，将此空值缓存到Redis中，并设置一定的缓存过期时间如60s。
  3. 布隆过滤器。即我们本文要学习的内容，可以借助此数据结构提前将数据预热到布隆过滤器中以便后续筛选掉一定不存在的数据。

可能会有同学问，既然有方法1和2，我们为什么还需要布隆过滤器？我们知道，对于方法1而言，依赖于具体key的生成规则，如果不是id这种具有清晰规则的字段，则无法准确的作出判断，依旧有可能通过暴力的方式找出一个符合攻击要求的key。对于方法2而言，很明显，当在遭受恶意攻击时在某一时刻Redis中会存在大量的空值数据，从而影响缓存的性能，尽管设置一定的缓存过期时间可以缓解此问题造成的性能瓶颈，然而依旧不是那么的优雅。

下面让我们来了解一下布隆过滤器。

布隆过滤器，英文名称为Bloom Filter，是一位名叫Howard Bloom的老哥于1970年提出的二进制向量数据结构，我们可以把它看作由一系列随机映射函数和位数组而组成的数据结构。位数组中每个元素都只占1bit，并且每个元素不是0就是1。如果我们要申请500w个元素的位数组只需要占用5000000 / 8 bit = 625000 Byte ≈ 610 kb。布隆过滤器可以被用于邮箱黑名单的筛选、爬虫对网站url去重等场景，很显然，我们可以用它来帮助我们预防缓存穿透。

####布隆过滤器的实现原理如下所述：

  1. 当一个元素被添加到布隆过滤器中时，会使用各个预设的哈希函数进行计算得到的哈希值即为位数组的下标，然后将此下标的值置为1.
  2. 当判断一个元素是否已经存在时，对该元素进行哈希计算，如果哈希下标对应的值都是1则说明已经存在该元素，但凡有一个哈希函数计算为出的哈希坐标对应值为0，则说明该元素一定不存在。
    这里需要注意一下上文中第2点的描述，由于不同元素计算出的哈希值可能相同，并且在布隆过滤器中元素越多的情况下，这种误判的概率就会越大，因此我们可以根据实际情况调整布隆过滤器的大小避免哈希碰撞。但简而言之，虽然布隆过滤器判定某个元素存在可能存在小概率误判，但是当判定某个元素不存在时则该元素一定不存在，从风险角度来说布隆过滤器仍然是我们用来治理缓存穿透的好伙伴。

Redis数据库双写一致性问题 - 缓存删除失败

发表于 2020-01-05 更新于 2020-02-12 分类于 技术

Redis是一款基于内存的数据库，虽然在项目中我们常常称其为”缓存”，然而此处的缓存是基于内存的数据库，存在于RAM中，而并非我们操作系统语义范围内比内存存取速度更快的Cache，尽管如此，基于内存的Redis的读写性能也仍然远远高于高度依赖磁盘I/O的数据库存取，项目中我们常常使用Redis提高分布式系统处理高并发的能力。

假设我们现在有两个线程A和B，A更新数据库成功后删除Redis中的数据，B线程处理读取数据的请求时先访问Redis，如果不命中则访问数据库并将内容更新到Redis中。

这样的做法流程上看起来是没有问题的，然而当Redis删除失败的时候却无法保证数据的一致性。有些项目的做法是要么等到过期后接口查询主请求查询数据库后主动更新Redis，或者操作场务界面保存触发删除，这两种办法实时性不高，尤其是第二种界面删除的操作过度依赖于人工操作，无法及时的挽救线上故障。

可选的方案：

  1. A线程更新数据库，发送事件通知负责处理一致性的线程C；
  2. C线程负责订阅A线程发送的数据库更新消息，并使用从消息中获取到的key对Redis进行删除操作；
  3. 如果第2步中Redis删除失败，则将这些信息存入Redis中预初始化的list数据结构中；
  4. C线程重新从list中获得更新数据，重试删除，如果失败再重写回list中，直到Redis删除成功为止。

其中第4步如果考虑高可用场景可以对Redis开启持久化，以免Redis重启后内存丢失数据list不可用。
